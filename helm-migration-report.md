# Helm Migration Report

**Repository:** klusty-krab-home-lab  
**Date:** 2026-06-14

This report identifies which Argo CD applications are currently deployed using hand-written raw YAML manifests and assesses whether each one can be migrated to a Helm chart instead.

---

## Current State

### Already using Helm

These apps are already pulling a Helm chart; no migration needed.

| App | Chart | Repo |
|-----|-------|------|
| `plex` | `gabe565/plex` 0.5.1 | `ghcr.io/gabe565/charts` (OCI) |
| `longhorn` | `longhorn/longhorn` v1.9.0 | `https://charts.longhorn.io/` |
| `metallb` | `metallb/metallb` 0.15.2 | `https://metallb.github.io/metallb` |

Both `longhorn` and `metallb` use **Argo multi-source**: the Helm chart as primary source, plus a git path for supplementary CRs (`IPAddressPool`, `L2Advertisement`, dashboard `Service`). This pattern is the recommended approach for the apps listed below when cluster-specific resources (fixed LoadBalancer IPs, PersistentVolumes) need to live in git alongside the chart.

### Using Helm indirectly via k3s `HelmChart` CRs

These apps deploy `helm.cattle.io/v1 HelmChart` objects that are applied by the k3s embedded Helm controller. Argo CD syncs the CR definitions; the actual chart install is handled in-cluster. They are functionally equivalent to Helm deployments.

| App | Charts deployed |
|-----|----------------|
| `intel-gpu-dependencies` | `cert-manager`, `node-feature-discovery`, `intel-device-plugins-operator` |
| `traefik` | k3s built-in Traefik, configured via `HelmChartConfig` CR |

## Apps using raw YAML — migration assessment

### Tier 1 — Recommended: well-maintained charts exist

These apps have active, community-maintained Helm charts and are straightforward migration targets.

---

#### `adguard`

**Current setup:** Deployment + LoadBalancer Service (`10.0.0.241`, DNS UDP 53) + 2 Longhorn PVCs.

**Recommended chart:** [`k8s-at-home/adguard-home`](https://github.com/k8s-at-home/charts/tree/master/charts/stable/adguard-home) or `bjw-s/app-template` (OCI: `ghcr.io/bjw-s/helm/app-template`).

The bjw-s `app-template` is the more actively maintained option and is widely used in the home-lab community. The k8s-at-home chart is archived but still functional.

**Migration considerations:**
- Set `service.main.loadBalancerIP: 10.0.0.241` and `service.main.ports.dns.protocol: UDP` and `port: 53` in values.
- Map the two PVCs (`adguard-config`, `adguard-data`) to `persistence` blocks pointing to the same Longhorn `storageClass`.
- The `adguard-namespace.yaml` is replaced by `CreateNamespace=true` (already present on the Argo Application).

**Argo pattern to use:** Single Helm source (no supplementary git path needed — no extra CRs beyond what the chart creates).

---

#### `audiobookshelf`

**Current setup:** Deployment (image `advplyr/audiobookshelf:2.34.0`) + LoadBalancer (`10.0.0.235`) + 2 Longhorn PVCs + hostPath `/mnt/media/Audiobooks`.

**Recommended chart:** [`k8s-at-home/audiobookshelf`](https://github.com/k8s-at-home/charts/tree/master/charts/stable/audiobookshelf) or `bjw-s/app-template`.

**Migration considerations:**
- Set `service.main.loadBalancerIP: 10.0.0.235`.
- The hostPath for `/mnt/media/Audiobooks` requires a static PersistentVolume in git (same pattern as existing `media-smb` and `roms`). Use the **multi-source** pattern: Helm chart as source 1, git path with the static PV/PVC YAML as source 2.
- Map `persistence.config` → 2 Gi Longhorn, `persistence.audiobooks` → existing static PVC.

#### `calibre-web`

**Current setup:** Deployment (image `crocodilestick/calibre-web-automated`) + LoadBalancer (`10.0.0.234`) + Traefik `IngressRoute` (`calibre.home`) + 10 Gi Longhorn PVC + hostPath `/mnt/media/Books`.

**Recommended chart:** [`k8s-at-home/calibre-web`](https://github.com/k8s-at-home/charts/tree/master/charts/stable/calibre-web) or `bjw-s/app-template`.

**Note:** The current deployment uses the `calibre-web-automated` image rather than the upstream `linuxserver/calibre-web`. Make sure to keep `image.repository: crocodilestick/calibre-web-automated` in values.

**Migration considerations:**
- Set `service.main.loadBalancerIP: 10.0.0.234`.
- hostPath for `/mnt/media/Books` → static PV/PVC in git path (multi-source pattern).
- The Traefik `IngressRoute` CRD is not a standard `Ingress`, so it cannot be generated by the chart's built-in ingress support. Keep the `IngressRoute` manifest in the git-path source alongside the static PV.

#### `cloudflared`

**Current setup:** Deployment (2 replicas, `cloudflare/cloudflared:latest`) + `tunnel-token` Secret (external, not in git).

**Recommended chart:**
- **Official (simpler):** [`cloudflare/cloudflare-tunnel`](https://github.com/cloudflare/helm-charts/tree/main/charts/cloudflare-tunnel) — `helm repo add cloudflare https://cloudflare.github.io/helm-charts`. Note: the official chart pins to an older cloudflared app version (`2024.8.3`); override with `cloudflare.image.tag: latest` or a specific release tag in values.
- **Community (more active):** [`community-charts/cloudflared`](https://community-charts.github.io/docs/charts/cloudflared/usage) — `helm repo add community-charts https://community-charts.github.io/helm-charts`.

**Migration considerations:**
- This is the easiest migration: the current Deployment is nearly identical to what the chart produces.
- The `tunnel-token` Secret is already external; pass it via `env.TUNNEL_TOKEN.valueFrom.secretKeyRef` in values (both charts support this).
- 2-replica deployment is configurable via `replicaCount: 2`.
- No persistent storage or special networking needed.

#### `homebridge`

**Current setup:** Deployment (`hostNetwork: true`, `homebridge/homebridge`) + LoadBalancer (`10.0.0.248`) + 1 Gi Longhorn PVC.

**Recommended chart:**
- `bjw-s/app-template` — most flexible, supports `hostNetwork: true`.
- [`truecharts/homebridge`](https://charts.truecharts.org/) — purpose-built but heavier (TrueCharts dependency stack).
- [`espresso-lab/homebridge`](https://github.com/espresso-lab/helm-charts) (OCI: `oci://ghcr.io/espresso-lab/helm-charts/homebridge`) — lightweight alternative.

**Migration considerations:**
- `hostNetwork: true` is critical for HomeKit device discovery; set `podSpec.hostNetwork: true` in `app-template` values (or the chart's equivalent field).
- `hostNetwork` conflicts with setting `hostPort` — the existing Service with `10.0.0.248` may not be needed if Homebridge binds directly to the host IP. Verify current behavior before removing the Service.
- Persist `/homebridge` via Longhorn PVC — losing this volume severs all HomeKit pairings.

#### `syncthing`

**Current setup:** Deployment (`syncthing/syncthing:2.0.11`) + LoadBalancer (`10.0.0.245`, 3 ports: web/sync/discovery) + 5 PVCs/PVs (config on Longhorn; Saves, Dolphin, Cemu, Eden saves as hostPath PVs on `/mnt/retrogaming/*`).

**Recommended chart:** [`k8s-at-home/syncthing`](https://github.com/k8s-at-home/charts/tree/master/charts/stable/syncthing) or `bjw-s/app-template`.

**Migration considerations:**
- This is the most complex Tier 1 migration due to the 4 static hostPath PVs (`syncthing-pv`, `syncthing-dolphin-pv`, `syncthing-cemu-pv`, `syncthing-eden-pv`).
- Use the **multi-source** pattern: Helm chart + git path containing the 4 static PV YAMLs. The PVCs can then be referenced by name in chart values under `persistence`.
- Set `service.main.loadBalancerIP: 10.0.0.245` and expose all 3 ports.
- Pin image tag to `2.0.11` or later in values.

### Tier 2 — Possible, but low value

Helm charts exist for these apps (via `bjw-s/app-template`), but the current raw YAML is minimal enough that the migration adds complexity without meaningfully reducing maintenance burden.

---

#### `media-smb`

**Current setup:** Deployment (`ghcr.io/servercontainers/samba`) + LoadBalancer (`10.0.0.244`, TCP 445) + static PV/PVC pointing to hostPath `/mnt/media` (2500 Gi).

**Assessment:** The `servercontainers/samba` image uses a convention-driven config format that differs from mainstream Samba charts. The existing 4-file manifest is already minimal. Migration via `bjw-s/app-template` is possible but the values file would be roughly as long as the current raw YAML, with no reduction in complexity. **Low priority.**

## Recommended migration pattern

For all Tier 1 apps that mount hostPath volumes or expose MetalLB fixed IPs, the existing `longhorn` / `metallb` **multi-source** pattern is the right template:

```yaml
# apps/<app-name>.yaml (Argo CD Application)
spec:
  sources:
    # Source 1: Helm chart
    - repoURL: <chart-repo>
      chart: <chart-name>
      targetRevision: <version>
      helm:
        valueFiles:
          - $values/manifests/<app-name>/values.yaml
    # Source 2: git path for static PVs, IngressRoutes, etc.
    - repoURL: git@github.com:nourez/klusty-krab-home-lab.git
      targetRevision: HEAD
      ref: values   # reference alias for valueFiles above
    - repoURL: git@github.com:nourez/klusty-krab-home-lab.git
      targetRevision: HEAD
      path: manifests/<app-name>
      directory:
        recurse: true
  destination:
    namespace: <app-name>
    server: https://kubernetes.default.svc
  syncPolicy:
    automated:
      selfHeal: true
      prune: true
    syncOptions:
      - CreateNamespace=true
```

The `manifests/<app-name>/` directory then contains only the resources the chart cannot generate: static `PersistentVolume` objects, Traefik `IngressRoute` CRDs, and the Helm `values.yaml`.
